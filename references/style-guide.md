# Zu Article Style Guide

这份参考文件用于识别中文技术观点文里的 AI 写作痕迹：先找到模板骨架，再保留原文观点和事实边界，最后重写成更自然的技术表达。

这里有两层规则：

1. 把偏英文语境的写作痕迹改成中文提示词，例如把语法后缀类问题改成中文写作里更常见的 `句尾强行升华和结果套话`。
2. 融入用户自己的技术观点文要求：保留工程判断、观点归属、技术 caveat，不凭空新增事实，不写白皮书腔。

重要边界：下面的“改写后”只是说明方向。处理用户文章时，不能复制示例里的年份、数字、来源、案例、用户反馈，除非原文已经提供。

## 使用原则

### 先拆骨架，再润色

AI 味通常不是某个词，而是整段的生产方式：

- 先用宏大背景开场。
- 再抛一个假问题。
- 再用二元对比或三段式铺开。
- 最后用“标志着、彰显、进入新阶段”收尾。

处理时不要只换词。要先找到这段真正想说什么，再围绕真实主张重写。

### 不为了具体而编细节

很多去 AI 味示例会把空泛表述改成具体事实，这个方向是对的，但不能在真实文章里编事实。

可以把：

```markdown
这个项目展示了团队在不断演变的技术格局中的关键作用。
```

改成：

```markdown
这个项目真正值得写的，是它解决了什么问题，以及这些问题为什么以前解决不了。
```

如果原文没有具体功能、用户反馈或数据，不要补成“新增了批处理、快捷键和离线模式”。

### 技术观点文的核心节奏

用户文章最适合这个顺序：

1. 直接给判断。
2. 说明为什么这个判断成立。
3. 写清楚边界和风险。
4. 落到产品、工程或组织交付。

不要把文章写成概念百科、行业报告或中立综述。

## 中文 AI 写作痕迹模式库

### 1. 空泛意义升华

识别：

- `标志着`
- `见证了`
- `体现了`
- `证明了`
- `奠定了基础`
- `关键时刻`
- `不可磨灭的印记`
- `不断演变的格局`

问题：普通事实被写成宏大意义，读者看不到具体信息。

改写前：

```markdown
加泰罗尼亚统计局于 1989 年正式成立，标志着西班牙区域统计演变史上的关键时刻。这一举措是西班牙全国范围内更广泛运动的一部分，旨在分散行政职能并加强区域治理。
```

改写后：

```markdown
加泰罗尼亚统计局成立于 1989 年，负责独立于西班牙国家统计局收集和发布区域统计数据。
```

技术文方向：

```markdown
弱：这套 Agent 架构标志着企业智能化进入新阶段。
好：这套 Agent 架构真正解决的是权限、状态保留和工具调用可观测性，模型能力反而不是最难的部分。
```

### 2. 背书堆砌

识别：

- `被多家媒体报道`
- `行业广泛关注`
- `专家高度评价`
- `在社交媒体上引发热议`

问题：只堆背书，不写具体观点。

改写前：

```markdown
她的观点被《纽约时报》、BBC、《金融时报》和《印度教徒报》引用。她在社交媒体上拥有活跃的存在，拥有超过 50 万粉丝。
```

改写后：

```markdown
在 2024 年《纽约时报》的采访中，她认为 AI 监管应该关注结果而不是方法。
```

真实文章里要注意：如果没有具体来源，不要编媒体和日期。可以改成“这里需要具体来源，否则这句只能删掉”。

### 3. 句尾强行升华和结果套话

中文里常见表现不是语法后缀，而是句尾硬接一串总结词。

识别：

- `确保...`
- `彰显...`
- `反映...`
- `象征...`
- `展示...`
- `凸显...`
- `为...提供支撑`

问题：后半句没有新增信息，只是在给前半句贴意义标签。

改写前：

```markdown
寺庙的蓝色、绿色和金色色调与该地区的自然美景产生共鸣，象征着德克萨斯州的蓝帽花、墨西哥湾和多样化的德克萨斯州景观，反映了社区与土地的深厚联系。
```

改写后：

```markdown
寺庙使用蓝色、绿色和金色。建筑师表示，这些颜色是为了呼应当地的蓝帽花和墨西哥湾海岸。
```

技术文方向：

```markdown
弱：这套工具链提升了研发效率，彰显了 AI 在软件工程中的关键价值。
好：这套工具链省掉了重复检索和上下文整理，但代码审查、风险判断和上线责任仍然在人这边。
```

### 4. 宣传和广告腔

识别：

- `无缝`
- `直观`
- `强大`
- `充满活力`
- `令人叹为观止`
- `开创性`
- `必备`
- `重要抓手`
- `战略高地`

问题：像产品宣传，不像技术判断。

改写前：

```markdown
坐落在埃塞俄比亚贡德尔地区令人叹为观止的区域内，Alamata Raya Kobo 是一座充满活力的城镇，拥有丰富的文化遗产和迷人的自然美景。
```

改写后：

```markdown
Alamata Raya Kobo 是埃塞俄比亚贡德尔地区的一座城镇，以每周集市和 18 世纪教堂而闻名。
```

技术文方向：

```markdown
弱：这是一个无缝、直观、强大的 Agent 工作流平台。
好：它把任务拆解、工具调用、状态保存和结果回放放到同一条链路里，方便团队追踪一次 Agent 运行到底做了什么。
```

### 5. 模糊归因

识别：

- `行业报告显示`
- `专家认为`
- `观察者指出`
- `有观点认为`
- `多个来源表明`
- `业内普遍认为`

问题：观点没有归属，读者无法判断可信度。

改写前：

```markdown
由于其独特的特征，浩来河引起了研究人员和保护主义者的兴趣。专家认为它在区域生态系统中发挥着至关重要的作用。
```

改写后：

```markdown
根据中国科学院 2019 年的调查，浩来河支持多种特有鱼类。
```

技术文方向：

```markdown
弱：很多专家认为 Agent 会成为企业软件的新入口。
好：如果没有明确来源，这句话应该删掉。作者真正想表达的可以写成：企业级 Agent 的难点在权限、流程和审计要求。
```

### 6. 套路化挑战与未来展望

识别：

- `尽管面临若干挑战`
- `尽管存在这些挑战`
- `未来展望`
- `仍将继续蓬勃发展`
- `未来可期`

问题：最后一段看起来完整，但没有给出真实结论。

改写前：

```markdown
尽管工业繁荣，Korattur 面临着城市地区典型的挑战，包括交通拥堵和水资源短缺。尽管存在这些挑战，凭借其战略位置和正在进行的举措，Korattur 继续蓬勃发展，成为钦奈增长不可或缺的一部分。
```

改写后：

```markdown
2015 年三个新 IT 园区开业后，交通拥堵加剧。市政公司于 2022 年启动了雨水排水项目，以解决反复发生的洪水。
```

技术文方向：

```markdown
弱：尽管 Agent 落地仍面临挑战，但未来发展空间巨大。
好：Agent 落地最大的瓶颈不在 Demo，而在长期运行后的成本、权限、评测和责任边界。
```

### 7. AI 高频词

识别：

- `此外`
- `值得注意的是`
- `深入探讨`
- `持续演进`
- `复杂格局`
- `关键作用`
- `重要意义`
- `宝贵`
- `充满活力`
- `赋能`
- `闭环`
- `降本增效`

问题：这些词经常成组出现，让文章变得像模板。

改写前：

```markdown
此外，索马里菜肴的一个显著特征是加入骆驼肉。意大利殖民影响的持久证明是当地烹饪格局中广泛采用意大利面，展示了这些菜肴如何融入传统饮食。
```

改写后：

```markdown
索马里菜肴还包括骆驼肉，被认为是一种美味。在意大利殖民期间引入的意大利面菜肴仍然很常见，尤其是在南部。
```

技术文方向：

```markdown
弱：Agent 通过闭环工具链赋能研发流程，实现降本增效。
好：Agent 真正省下来的，是人在检索、整理上下文和重复执行上的时间。代码质量和上线风险仍然要单独评估。
```

### 8. 绕开简单判断

识别：

- `作为`
- `代表`
- `标志着`
- `充当`
- `具备...能力`
- `提供了一个...`

问题：明明可以用“是、有、能、不能”，却绕成大句。

改写前：

```markdown
Gallery 825 作为 LAAA 的当代艺术展览空间。画廊设有四个独立空间，拥有超过 3000 平方英尺。
```

改写后：

```markdown
Gallery 825 是 LAAA 的当代艺术展览空间。画廊有四个房间，总面积 3000 平方英尺。
```

技术文方向：

```markdown
弱：Skill 作为 Agent 能力扩展的重要载体，具备沉淀专家经验的能力。
好：Skill 是给 Agent 用的操作手册。它适合沉淀稳定流程，不适合承载所有业务知识。
```

### 9. 否定式排比

识别：

- `不是...而是...`
- `而不是`
- `不仅仅是...更是...`
- `这不只是...它还是...`

问题：看起来有判断，实际是模板对比。

改写前：

```markdown
这不仅仅是一次更新，而是我们思考生产力方式的革命。
```

改写后：

```markdown
这次更新新增了批处理、键盘快捷键和离线模式。
```

技术文方向：

```markdown
弱：Agent 不是一个工具，而是一套新的工作方式。
好：Agent 是否成立，取决于它是否稳定接住一条具体工作流。
```

### 10. 三件套结构

识别：

- `主要有三点`
- `现在比的是三件事`
- 三个并列形容词：`无缝、直观、强大`
- 每段都按三项展开

问题：为了显得完整强行凑三项。

改写前：

```markdown
活动包括主题演讲、小组讨论和社交机会。与会者可以期待创新、灵感和行业洞察。
```

改写后：

```markdown
活动包括演讲和小组讨论。会议之间还有非正式交流时间。
```

技术文方向：

```markdown
弱：现在比的是三件事：模型、工具和生态。
好：我会重点看两件事：模型调用工具是否稳定，这套调用链是否可以评测和追责。
```

### 11. 用户人工标记的不允许出现句式

这些句式不一定都能通过通用规则识别，但用户已经明确判断为 AI 味很浓。除非用户原文是引用、吐槽或分析这些句式，否则正文里不要保留。

#### 大家最容易盯着...

不允许出现：

```markdown
大家最容易盯着模型名、参数、榜单和发布会。
```

问题：这是很常见的 AI 开场框架，用“大家都在看错地方”制造观点姿态。

处理方向：直接写作者要讨论的判断，不要用“大家最容易...”做铺垫。

#### 稳不稳、能不能、会不会密集叠加

不允许出现：

```markdown
接口稳不稳，工具调用能不能跑长链路，成本能不能算清楚，模型升级会不会把已有工程掀翻。
```

问题：连续反问式判断太整齐，像模型在凑“真实问题清单”。

处理方向：改成更普通的判断句或拆成短句，避免四个相同句式并排。

#### 我的判断很直接、很扎心、很现实

不允许出现：

```markdown
我的判断很直接。
我的判断很扎心。
我的判断很现实。
```

问题：这是刻意模仿人类表达，出现频率高后反而更像 AI。

处理方向：可以保留真实判断，但不要先给判断贴“直接、扎心、现实”的标签。

#### 相比较...我更关心这几个问题

不允许出现：

```markdown
相比较模型参数，我更关心这几个问题。
```

问题：这是固定转场句式，容易把文章拉回模板。

处理方向：直接进入具体问题，或用一句更自然的上下文承接。

#### 可接入、可上线、可控成本

不允许出现：

```markdown
这说明它瞄准的是企业可接入、可上线、可控成本的生产系统。
```

问题：这类 `可 X、可 Y、可 Z` 排比太密，像产品白皮书。

处理方向：如果这些确实是重点，拆成更具体的约束：接口、部署、成本、迁移、治理，各自说明一句。

### 12. 同义词循环

识别：同一概念频繁换名，导致读者不知道是不是同一件事。

改写前：

```markdown
主人公面临许多挑战。主要角色必须克服障碍。中心人物最终获得胜利。英雄回到家中。
```

改写后：

```markdown
主人公面临许多挑战，但最终获得胜利并回到家中。
```

技术文方向：

```markdown
弱：智能体、Agent 系统、自主执行体、任务代理在同一段里反复切换。
好：选一个词并保持稳定。除非你明确区分这些概念。
```

### 13. 虚假跨度

识别：

- `从 A 到 B`
- `从 X 走向 Y`
- `从 X 转向 Y`
- `覆盖从...到...的全链路`

问题：A 和 B 不是同一个尺度，跨度只是为了显得宏大。

改写前：

```markdown
我们穿越宇宙的旅程将我们从大爆炸的奇点带到宏伟的宇宙网，从恒星的诞生和死亡到暗物质的神秘舞蹈。
```

改写后：

```markdown
这本书涵盖了大爆炸、恒星形成和当前关于暗物质的理论。
```

技术文方向：

```markdown
弱：AI 应用已经从模型调用走向业务闭环。
好：模型调用只是开始。难点在业务状态、权限、工具结果和人工确认怎么串起来。
```

### 14. 破折号强行制造力度

识别：一段里连续使用 `——`，用来制造节奏或揭示感。

改写前：

```markdown
这个术语主要由荷兰机构推广——而不是由人民自己。你不会说“荷兰，欧洲”作为地址——但这种错误标记仍在继续——即使在官方文件中。
```

改写后：

```markdown
这个术语主要由荷兰机构推广，而不是由人民自己。你不会说“荷兰，欧洲”作为地址，但这种错误标记在官方文件中仍在继续。
```

技术文可以保留少量破折号，但不能靠它撑语气。

### 15. 粗体滥用

识别：每个术语都加粗，或者每个列表项都用粗体开头。

改写前：

```markdown
它融合了 **OKR**、**KPI** 和视觉战略工具，如 **商业模式画布** 和 **平衡计分卡**。
```

改写后：

```markdown
它融合了 OKR、KPI 和视觉战略工具，如商业模式画布和平衡计分卡。
```

用户文章里，粗体只用于真正的判断句。

### 16. 粗体小标题列表

识别：

```markdown
- **用户体验：** 用户体验通过新界面得到显著改善。
- **性能：** 性能通过优化算法得到增强。
- **安全性：** 安全性通过端到端加密得到加强。
```

改写后：

```markdown
更新改进了界面，通过优化算法加快了加载时间，并添加了端到端加密。
```

技术文方向：如果列表项不是独立模块，就合并成一句或改成普通 bullet。

### 17. 标题大而空

中文标题更常见的问题是抽象、宏大、没有判断。

需要警惕：

```markdown
## Agent 技术架构与未来发展趋势
```

处理方向：不要在参考规则里预设“好标题”。改标题时只保留原文已经成立的判断，并检查标题是否比正文更激进、更宏大或更像口号。

### 18. emoji 装饰

识别：用 emoji 装饰标题、列表或步骤。

改写前：

```markdown
🚀 启动阶段：产品在第三季度发布
💡 关键洞察：用户更喜欢简单
✅ 下一步：安排后续会议
```

改写后：

```markdown
产品在第三季度发布。用户研究显示，用户更喜欢简单界面。下一步是安排后续会议。
```

技术观点文默认不用 emoji。

### 19. 引号和标点混乱

识别：

- 中文正文里混用 `" "`、`“ ”`、`「 」`
- 半角冒号和全角冒号混用
- 列表标点不统一

处理：按文章既有风格统一。不要为了统一破坏代码、命令、术语。

### 20. 聊天回复残留

识别：

- `当然`
- `希望这对你有帮助`
- `如果你需要我可以继续`
- `这是一个很好的问题`
- `下面是`

改写前：

```markdown
这是法国大革命的概述。希望这对您有帮助！如果您想让我扩展任何部分，请告诉我。
```

改写后：

```markdown
法国大革命始于 1789 年，当时财政危机和粮食短缺导致了广泛动荡。
```

### 21. 知识截止日期免责声明

识别：

- `截至我的知识截止日期`
- `根据我最后的训练更新`
- `基于可用信息`
- `虽然具体细节有限`

改写前：

```markdown
虽然关于公司成立的具体细节在现成资料中没有广泛记录，但它似乎是在 20 世纪 90 年代的某个时候成立的。
```

改写后：

```markdown
根据注册文件，该公司成立于 1994 年。
```

真实文章里：如果没有来源，不要补具体年份。写成“这里需要查证，不能用模型免责声明代替来源”。

### 22. 讨好语气

识别：

- `好问题`
- `你说得完全正确`
- `这是一个非常重要的话题`
- `非常棒的观点`

改写前：

```markdown
好问题！您说得完全正确，这是一个复杂的话题。关于经济因素，这是一个很好的观点。
```

改写后：

```markdown
你提到的经济因素在这里是相关的。
```

用户文章里默认不需要讨好读者。

### 23. 填充短语

常见替换方向：

| 弱表达               | 更直接             |
| -------------------- | ------------------ |
| 为了实现这一目标     | 为了做到这一点     |
| 由于下雨的事实       | 因为下雨           |
| 在这个时间点         | 现在               |
| 系统具有处理的能力   | 系统可以处理       |
| 值得注意的是数据显示 | 数据显示           |
| 在某种程度上可以说   | 可以说，或直接删除 |

技术文里能删就删。删不掉再改短。

### 24. 过度限定

识别：

- `可能会潜在地`
- `在某种程度上`
- `也许可以认为`
- `不排除存在`

改写前：

```markdown
可以潜在地可能被认为该政策可能会对结果产生一些影响。
```

改写后：

```markdown
该政策可能会影响结果。
```

技术文方向：

```markdown
弱：Agent 在某种程度上可能会对研发流程产生一定影响。
好：Agent 会影响研发流程里的检索、上下文整理和重复执行，但不直接替代代码责任。
```

### 25. 模糊乐观结尾

识别：

- `未来可期`
- `激动人心的时代`
- `继续追求卓越`
- `迈出重要一步`
- `让我们拭目以待`

改写前：

```markdown
公司的未来看起来光明。激动人心的时代即将到来，他们继续追求卓越的旅程。这代表了向正确方向迈出的重要一步。
```

改写后：

```markdown
该公司计划明年再开设两个地点。
```

技术文方向：

不要用“未来值得期待、应积极拥抱”收尾。也不要再改成一个判断前缀后面跟一串排比问题。更稳的处理方式是回到原文已有结论：如果原文有明确风险，就保留风险；如果原文没有结论，就删掉乐观收尾。

## 发布前检查

- 是否先拆掉 AI 骨架，再做结构整理。
- 是否保留了用户原本的判断和技术边界。
- 是否删除或重写了 AI 写作痕迹：空泛升华、宣传腔、模糊归因、句尾升华、三件套、聊天残留、乐观套话、人工感叠词、刻意拟人口吻。
- 是否没有新增未经原文支撑的事实、数字、案例、来源。
- 章节编号是否连续。
- 粗体是否只用于核心判断。
- 总结是否短，是否避免把全文重新概括一遍。
